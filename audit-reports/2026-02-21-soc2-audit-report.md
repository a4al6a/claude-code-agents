### FINDING-001: Hardcoded Credentials in Example Files Committed to Repository

**Severity**: Critical
**Confidence**: High
**Risk Score**: 75 (Severity 5 x Likelihood 3 x Impact 5)
**Location**: `code-smell-detector/examples/legacy-refactoring/UserManager.java:27` and `code-smell-detector/examples/legacy-refactoring/UserManager.java:48`; duplicated at `refactoring-expert/examples/legacy-refactoring/UserManager.java:27` and `:48`
**Framework Mapping**: SOC 2 CC6.1 -- Logical Access Security

**Evidence**:
```java
// Line 27:
private String SMTP_PASS = "hardcodedpassword123"; // TODO: Move to config
// Line 48:
"password123" // Another hardcoded password!
```
These files exist in the git history and are present in two separate directories. While they are pedagogical examples demonstrating code smells, they are fully committed Java source files containing string literals that match credential patterns.

**Criteria**:
SOC 2 CC6.1 requires that logical access security mechanisms prevent unauthorized access. Hardcoded credentials in source control -- even in example code -- create a risk if cloned, forked, or deployed without context. OWASP classifies hardcoded secrets as a critical vulnerability.

**Impact**:
If this example code were ever executed or adapted without removing the credentials, it would expose SMTP and database access. The credentials "hardcodedpassword123" and "password123" are trivially guessable. Repository cloners may not realize these are intentional anti-patterns.

**Recommendation**:
Replace hardcoded credential values with clearly non-functional placeholders (e.g., `<REDACTED-EXAMPLE>` or `REPLACE_ME`). Add a prominent warning comment and a `.env.example` file pattern. Consider adding a pre-commit hook with secret scanning (e.g., `detect-secrets`, `gitleaks`).

**Effort**: S

### FINDING-008: Installer Uses rm -rf Without Safety Guards

**Severity**: Medium
**Confidence**: High
**Risk Score**: 27 (Severity 3 x Likelihood 3 x Impact 3)
**Location**: `install.sh:257` and `install.sh:321`
**Framework Mapping**: SOC 2 CC8.1 -- Change Management; SOC 2 PI1.1 -- Processing Integrity

**Evidence**:
```bash
# Line 257:
rm -rf "$skill_target"
# Line 321:
rm -rf "$skill_target"
```
The `$skill_target` variable is constructed as `$CLAUDE_SKILLS_DIR/$skill_name` where `$CLAUDE_SKILLS_DIR` defaults to `$HOME/.claude/skills`. While the variable construction appears safe (using predefined constants), `rm -rf` on user home directory paths has inherent risk if variables are ever empty or malformed.

**Criteria**:
SOC 2 PI1.1 requires that processing is complete, valid, accurate, and timely. Destructive file operations in user home directories should include safety guards.

**Impact**:
If `$skill_target` were to evaluate to an unintended path due to variable expansion failure, `rm -rf` could delete arbitrary directories. The `set -e` at line 7 provides some protection, but does not guard against valid but incorrect paths.

**Recommendation**:
Add defensive checks before `rm -rf`: validate that `$skill_target` contains expected path components, is not empty, and is not a system directory. Consider using `rm -rf "${skill_target:?Variable not set}"` pattern.

### FINDING-009: No Input Validation on Installer Script Arguments

**Severity**: Medium
**Confidence**: Medium
**Risk Score**: 18 (Severity 3 x Likelihood 2 x Impact 3)
**Location**: `install.sh:345-411`
**Framework Mapping**: SOC 2 CC6.6 -- Security Against External Threats

**Evidence**:
The `main()` function (lines 345-411) accepts arbitrary command-line arguments. The `specific_agent` variable (line 367-368) is set from user input without validation or sanitization:
```bash
*)
    if [[ -z "$specific_agent" ]]; then
        specific_agent="$1"
    fi
```
This value is then passed to `find_agent_def()` which iterates through a whitelist array, providing implicit validation. However, the input is also echoed back to the user in error messages (line 207) without sanitization.

**Criteria**:
SOC 2 CC6.6 requires that the entity implements controls to prevent or detect and act upon the introduction of unauthorized or malicious software. Input validation is a fundamental secure coding practice.

**Impact**:
Low practical risk due to the whitelist comparison in `find_agent_def()`, but the pattern of accepting unvalidated input in shell scripts is a minor code quality concern. Terminal escape sequences in agent names could potentially be used for display manipulation.

**Recommendation**:
Add explicit input sanitization (alphanumeric and hyphens only) for `specific_agent` before processing. Sanitize output in error messages.

**Effort**: S
